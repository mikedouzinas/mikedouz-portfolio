**Status**: Planned

## Overview

A Claude Code command in the-mv-vault that reads processed Spotify music moments from the portfolio repo and cross-references them with vault content (daily journal, projects, academics, life events) to build a living "Music Timeline" document. The document connects songs to life context — what was happening during each listening burst.

## How It Works

### Data Flow

```
mikedouz-portfolio (already built)
  ├── npm run spotify:analyze → music-moments.json (532 moments, 4 years)
  ├── npm run spotify:refresh → pulls recent plays from Spotify API
  └── src/data/spotify/music-moments.json (source of truth)

the-mv-vault (to be built)
  └── /music-timeline command
      ├── Reads music-moments.json from portfolio path
      ├── Scans vault for content matching each moment's date range
      ├── Builds/updates 02_Areas/Leisure & Curiosity/Music Timeline.md
      └── Two modes: full catchup + incremental update
```

### No Duplicated Work

- The vault NEVER calls the Spotify API or runs Kleinberg
- It reads the already-processed `music-moments.json` from the portfolio repo
- Portfolio repo handles: raw data → normalization → Kleinberg → enrichment → JSON
- Vault handles: JSON → life context matching → Music Timeline document

## The Command: `/music-timeline`

### Location
`the-mv-vault/.claude/commands/music-timeline.md` (private vault, not apollo-vault)

### Two Modes

#### 1. Full Catchup (`/music-timeline catchup`)
First-time run or periodic full refresh. Processes ALL moments across the entire 4-year history.

1. Read `music-moments.json` from portfolio path
2. For each moment's date range, scan the vault for matching content:
   - `06_Metadata/Daily Journal/YYYY-MM-DD*.md` — daily entries overlapping the date range
   - `02_Areas/Academics/` — lecture notes, homework with dates in filenames
   - `01_Projects/` — project notes with date references
   - `02_Areas/` — any area content with matching dates
   - `04_Archive/` — archived projects (Barcelona abroad, etc.) with date context
3. For each moment, extract relevant snippets/themes from matching vault content
4. Build the Music Timeline document organized by time period (reverse chronological)
5. Each entry: song info + date range + hot days + what was happening in life

#### 2. Incremental Update (`/music-timeline update`)
Bi-weekly run after `npm run spotify:refresh && npm run spotify:analyze` updates the JSON.

1. Read `music-moments.json` — check for moments newer than the last entry in the Music Timeline
2. Only process new/changed moments
3. Append to the top of the Music Timeline document
4. Cross-reference only recent vault content (last 2-3 weeks)

### How Life Context Is Extracted

The command should scan vault content within each moment's date range and look for:

1. **Daily journal entries** — the richest source. What Mike was thinking/doing that day
2. **Academic submissions** — homework dates show what classes were active
3. **Project activity** — files created/modified in that window
4. **Life events** — any notable entries (meetings, decisions, milestones)
5. **Emotional/reflective content** — from Self-Knowledge area, if present

The command uses its own judgment to write 1-3 sentence summaries connecting the music to the life context. It should be natural and personal, not robotic.

## Integration Points

### Keeping Data Fresh

The portfolio repo's `npm run spotify:refresh` + `npm run spotify:analyze` pipeline runs every ~2 weeks. This updates `music-moments.json` with new moments. The vault's `/music-timeline update` then reads the updated JSON and adds new entries.

**Workflow:**
```bash
# In portfolio repo (or automated):
cd ~/Downloads/Dev/mikedouz-portfolio
npm run spotify:rebuild   # refresh + analyze

# In vault (manual or as part of weekly review):
/music-timeline update
```

This could be integrated into the vault's existing `/weekly-synthesis` command.

### Cross-Linking

The Music Timeline document can link to other vault notes:
- `[[2026-01-08]]` — link to daily journal
- `[[GREK 7B - Intermediate Modern Greek]]` — link to active courses
- `[[Project Olympus]]` — link to active projects

This makes the music timeline a connected node in the vault's knowledge graph.

## Implementation Notes

### For the Vault Command

The command (`music-timeline.md`) should:
1. Define the portfolio data path as a constant
2. Use `Read` tool to load `music-moments.json`
3. Use `Glob` to find vault files matching date ranges
4. Use `Read` to scan matching files for context
5. Use `Write` to create/update the Music Timeline document

### Heat Colors for the Document

Use the same color scale as the website:
- 🔴 Red — on fire (10+ plays/week)
- 🟠 Orange — very hot (7-9 plays/week)
- 🟡 Yellow — hot (5-6 plays/week)
- 🟢 Green — warm (3-4 plays/week)
- 🔵 Blue — moderate (< 3 plays/week)

### What NOT to Do

- Don't call the Spotify API from the vault
- Don't re-run Kleinberg from the vault
- Don't duplicate the music-moments data into a vault file (reference the JSON)
- Don't make the command too chatty — the Music Timeline should read like a journal, not an AI report
